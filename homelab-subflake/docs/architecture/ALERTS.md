# Alerts Architecture

## Overview

Alerting uses native Prometheus alerting rules evaluated by Prometheus and
routed through AlertManager for notification dispatch. Each service owns its
alert definitions; the framework generates the Prometheus rules from them.

**Flow:**

```no
services/*/non-functional/alerts.nix
  → framework generates Prometheus alerting rules
    → Prometheus evaluates, fires to AlertManager
      → AlertManager routes → ntfy-sh / Telegram
```

## Alert Levels

Numeric severity levels, used for routing and inhibition:

| Level         | Num | Notes                                      |
|---------------|-----|--------------------------------------------|
| Emergency     |  21 | Wake-up-now; very few things qualify       |
| Alert         |  19 |                                            |
| Critical      |  18 |                                            |
| Error         |  17 |                                            |
| Warning       |  13 |                                            |
| Notice        |  10 |                                            |
| Informational |   9 |                                            |
| Debug         |   5 |                                            |

**Routing:**

- Emergency → Telegram (loud)
- Everything else → ntfy-sh; vibration/sound if `_alertLevelNum > 17`, silent otherwise

## Service Alert Definition

Each service that wants alerts provides a `non-functional/alerts.nix`:

```nix
# services/ntp/non-functional/alerts.nix
{
  Emergency = [
    {
      title = "NTP is not synchronized";
      expr = "ntp_system_leap_indicator == 3";
      description = "Leap indicator is 3 (alarm/unsync)";
    }
  ];
  Warning = [
    {
      title = "High NTP offset";
      expr = "abs(ntp_source_offset_seconds) > 0.05";
    }
    {
      title = "High NTP send error rate";
      expr = "rate(ntp_server_response_send_errors_total[5m]) > 0";
    }
  ];
}
```

Referenced from `manifest.nix`:

```nix
observability.alerts.prometheusImpl = ./non-functional/alerts.nix;
```

### Schema

```nickel
AlertsDefinition :: {
  <Level>? : [AlertRule]
}

Level :: "Emergency" | "Alert" | "Critical" | "Error" | "Warning"
       | "Notice" | "Informational" | "Debug"

AlertRule :: {
  title              : String   # Alert name; becomes the Prometheus `alert:` field
  expr               : String   # PromQL expression; non-empty result = firing
  description?       : String   # Populates `annotations.description`
  for?               : String   # Pending duration before firing; defaults by level (see below)
  grafanaDashboardId?: String   # If set, a deep link to this dashboard is embedded in the notification
}
```

**`for` defaults by level** (override per alert with `for = "10m"`):

| Level                      | Default `for` |
|----------------------------|---------------|
| Emergency                  | `0m`          |
| Alert, Critical, Error     | `2m`          |
| Warning and below          | `5m`          |

**No Data behavior:** absent metrics produce no alert. Use `absent()` in the
`expr` if a missing metric should itself be alertable:

```nix
expr = "absent(ntp_system_leap_indicator) or ntp_system_leap_indicator == 3";
```

### Manifest Field

```nix
observability.alerts = {
  prometheusImpl : Path;   # Prometheus alerting rules (evaluated by Prometheus)
  # grafanaImpl  : Path;   # Future: Grafana-evaluated alerts for non-prometheus data
}
```

## Framework-Generated Labels

When generating Prometheus rules the framework automatically injects labels
that services do not need to declare:

| Label           | Value                        | Source                  |
|-----------------|------------------------------|-------------------------|
| `alertLevel`    | e.g. `Warning`               | severity bucket key     |
| `_alertLevelNum`| e.g. `"13"`                  | level numeric map       |
| `resource`      | `srv:<serviceName>`          | manifest service name   |
| `host`          | e.g. `host-a`                | data-flake `onHost`     |
| `service`       | e.g. `ntp`                   | manifest service name   |

For multi-instance services each instance gets its own `host` value from
data-flake (e.g. `dns-1` → `host-a`, `dns-2` → host-b).

## Host Down

A `HostDown` Prometheus rule is autogenerated per host. Its severity is
derived from the highest `_alertLevelNum` across all alerts defined by services
running on that host. No per-host configuration is needed.

```yaml
# sodium only runs low-priority services → Warning
- alert: HostDown
  expr: up{host="sodium"} == 0
  for: 2m
  labels:
    alertLevel: Warning
    _alertLevelNum: "13"
    host: sodium
    resource: "host:sodium"

# host-a runs services with Emergency alerts → Emergency
- alert: HostDown
  expr: up{host="host-a"} == 0
  for: 2m
  labels:
    alertLevel: Emergency
    _alertLevelNum: "21"
    host: host-a
    resource: "host:host-a"
```

Hosts with no services defining alerts default to `Warning`.

## Alert Storm Suppression

When a host goes down, AlertManager inhibits all service alerts for that host
— except Emergency, which always gets through:

```yaml
inhibit_rules:
  - source_matchers:
      - alertname = "HostDown"
    target_matchers:
      - resource =~ "^srv:.*"
      - alertLevel != "Emergency"
    equal: [host]
```

The `host` label is the correlation key injected at generation time. This means:

- One `HostDown` alert fires instead of a storm of service alerts
- If a service has an Emergency-level alert condition, it still fires regardless
- The `HostDown` alert itself carries the appropriate severity for that host
